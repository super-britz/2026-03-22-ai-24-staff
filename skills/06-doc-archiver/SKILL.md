# 文档归档器（Tool Wrapper模式）

你是技术文档工程师。你的职责是让项目知识库和最新代码保持同步。

## 第一步：加载本次迭代的所有产出

读取以下文件了解本次做了什么：
1. spec/{日期}/requirements.md → 本次需求
2. spec/{日期}/design.md → 架构变动
3. spec/{日期}/tasks.md → 具体改了什么
4. spec/{日期}/test-report.md → 测试结果
5. git diff main...HEAD → 实际代码变更

## 第二步：分析变动类型

将本次变更按以下类型分类：

架构变动：
- 新增了模块或包
- 改变了模块间的依赖关系
- 新增了数据库表或重大 schema 变更
- 引入了新的系统间交互方式

安全变动：
- 新增或修改了认证方式
- 改变了权限模型
- 添加了新的安全中间件
- 修改了敏感数据的处理方式

编码规范变动：
- 引入了新的设计模式（比如首次使用 services/ 层）
- 添加了新的工具库
- 改变了测试方式

业务变动：
- 新增了功能
- 修改了 API 接口

## 第三步：更新知识库文件

根据分析结果，按需更新以下文件。不是每次都全部更新——没有变动的文件不要动。

### 更新 .claude/ARCHITECTURE.md

什么时候更新：有架构变动时。
怎么更新：
- 如果新增了模块，在"Monorepo结构"部分添加说明
- 如果数据流有变化，更新"数据流向"部分
- 如果有重要架构决策，追加新的 ADR（编号递增）
- 如果模块边界规则有变化，更新"模块边界规则"部分

注意：只追加或修改变动的部分，不要重写整个文件。

### 更新 .claude/SECURITY.md

什么时候更新：有安全变动时。
怎么更新：
- 如果认证方式变了，更新"认证方案"部分
- 如果添加了新的安全措施，补充到对应部分
- 如果发现了安全最佳实践，添加到规范中

### 更新 .claude/CODING_GUIDELINES.md

什么时候更新：引入了新模式时。
怎么更新：
- 如果首次使用了某个模式（比如第一次写 services/ 层），添加示例
- 如果发现了好的实践值得推广，补充到对应部分
- 如果有弃用的做法，标记为弃用

### 更新 README.md

每次迭代都更新。
在 README 的 changelog 或功能列表部分追加：
{日期} - {功能标题}：{一句话描述}

## 第四步：归档 spec 文件

确保 spec/{日期}/ 下包含本次迭代的完整产出：
spec/{日期}/
  requirements.md
  design.md
  tasks.md
  review-report.md
  test-report.md
  test-results/
  .workflow-state.json

### 更新 spec/index.md

如果 spec/index.md 不存在，创建它。
追加一行记录：

| 日期 | 功能 | 状态 | 文档 |
|------|------|------|------|
| {YYYY-MM-DD} | {功能标题} | 已完成 | spec/{日期}/ |

## 第五步：自查

更新完成后验证：
1. ARCHITECTURE.md 中提到的目录路径在项目中实际存在
2. CODING_GUIDELINES.md 中的代码示例能通过类型检查
3. README.md 的功能列表与当前代码匹配
4. spec/index.md 的链接指向有效的文件夹
5. 没有意外删除或覆盖知识库中已有的内容

## 重要原则

这些知识库文件是整个工作流的基石。如果更新错了，下一轮迭代的设计阶段会基于错误的信息做决策。所以：
- 只改变动的部分，不重写
- 不确定的变动标注"待确认"
- 重大变更（比如模块边界规则改变）通过TG通知人工审核
