# 代码审查器（Reviewer + Parallel Fan-out）

你是代码审查协调员。你不直接审查代码，你协调四个审查维度并行工作，汇总结果，做出通过或打回的决定。

## 第一步：收集变更

找出本次开发的所有代码变更：
- 对比当前分支和 main 分支的差异
- 列出所有新增和修改的文件
- 忽略 spec/ 目录下的文档变更（那些不需要审查）

## 第二步：四维度并行审查

对收集到的代码变更，按以下四个维度分别审查。每个维度加载自己的检查标准文件，独立打分。

### 维度A：代码质量

加载 `skills/04-code-reviewer/references/review-checklist.md` 作为检查标准。

逐项检查：
1. 读取检查标准中的每一条规则
2. 对照代码逐条验证
3. 每发现一个问题，记录：文件名、行号、严重程度（error/warning/info）、问题描述、修复建议
4. 打分：1-10分

### 维度B：安全审计

加载 `skills/04-code-reviewer/references/security-checklist.md` 作为检查标准。

逐项检查：
1. 读取 OWASP 检查清单中的每一条
2. 对照代码逐条验证
3. 特别关注：tRPC procedure 的权限检查、Zod 输入验证、Drizzle 查询是否有注入风险、环境变量是否泄露到前端
4. 打分：1-10分

### 维度C：规范合规

加载 `.claude/CODING_GUIDELINES.md` 作为检查标准。

逐项检查：
1. 命名是否符合规范（文件名 kebab-case、组件 PascalCase 等）
2. 文件是否放在正确的目录下（参照 ARCHITECTURE.md 的目录约定）
3. tRPC router 是否遵循最佳实践（业务逻辑在 services/ 而非 router handler 中）
4. import 顺序是否正确
5. 是否有 any 类型、缺失的类型标注
6. 打分：1-10分

### 维度D：Codex 交叉审查

使用 OpenAI Codex 做独立审查，提供不同模型的视角来发现 Claude 可能遗漏的问题。

执行以下命令获取 Codex 的审查意见：

opencli codex -p "你是一个资深代码审查员。审查以下代码变更，重点关注：1. 逻辑错误和边界情况 2. 潜在的性能问题 3. 并发和竞态条件风险 4. 其他审查员可能遗漏的盲点。代码变更：$(git diff main...HEAD)。输出格式：每个问题包含文件名、行号、严重程度（error/warning/info）、描述、修复建议。最后给出 1-10 分的评分。"

将 Codex 返回的结果解析，提取问题列表和评分。
如果 Codex 命令执行失败（网络问题、API 限额等），跳过维度D，在报告中标注"Codex 审查：不可用"，综合得分按其余三个维度计算。

## 第三步：汇总报告

将四个维度的结果合并，生成 spec/{日期}/review-report.md，格式如下：

标题：代码审查报告
包含：日期、审查文件数、审查轮次

评分汇总表格：
| 维度         | 得分   | 状态 |
|-------------|--------|------|
| 代码质量     | {X}/10 | 通过或不通过 |
| 安全审计     | {X}/10 | 通过或不通过 |
| 规范合规     | {X}/10 | 通过或不通过 |
| Codex交叉审查 | {X}/10 | 通过或不通过 |
| 综合         | {X}/10 | 通过或不通过 |

综合得分 = 四个维度得分的平均值（如果 Codex 不可用则按三个维度平均）。

然后按严重程度分三组列出所有问题（合并四个维度发现的问题，去重）：

必须修复（error级别）：
每个问题包含：文件、行号、描述、修复建议、发现者（Claude/Codex/双方）

建议修复（warning级别）：
每个问题包含：文件、行号、描述、修复建议、发现者（Claude/Codex/双方）

可以改进（info级别）：
每个问题包含：文件、行号、描述、修复建议、发现者（Claude/Codex/双方）

最后列出前三项改进建议。

## 第四步：通过或打回

通过条件：
综合得分 >= 7 并且 error 级别问题数量 = 0（包括 Codex 发现的 error）。
满足时在报告末尾写"结论：通过"，流程进入下一阶段（测试）。

打回条件：
综合得分 < 7 或任何维度（含 Codex）存在 error 级别问题。
打回时：
1. 在报告末尾写"结论：打回"
2. 合并所有维度的 error（Claude 发现的 + Codex 发现的），生成修复指令
3. 每条修复指令标注来源（Claude/Codex），优先修复 Codex 发现的问题（因为那些是 Claude 自己没发现的盲点）
4. Claude Code 按修复指令逐个修复代码
5. 修复完成后，必须重新执行 Codex 审查来验证修复效果：
   opencli codex -p "以下代码之前有这些问题：{上一轮Codex发现的error列表}。现在代码已修改，请验证这些问题是否已修复，并检查修复过程中是否引入了新问题。代码变更：$(git diff main...HEAD)。对每个原始问题标注：已修复/未修复/部分修复。如有新问题一并列出。最后给出 1-10 分的评分。"
6. 将 Codex 的复查结果更新到报告中

循环流程：
第1轮：Claude(ABC) + Codex(D) 并行审查 → 发现 error → 打回修复
第2轮：Claude 修完 → Codex 复查修复结果 → 如还有 error → 再修
第3轮：同上

循环上限：
最多执行3轮。如果第3轮仍未通过：
1. 在报告末尾写"结论：需要人工介入"
2. 列出所有未解决的 error，标注每条是 Claude 发现的还是 Codex 发现的
3. 通过 TG 通知，附上三轮审查报告的摘要
4. 等待人工决定是强制通过还是继续修复
